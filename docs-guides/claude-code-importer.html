●  reading your Claude Code sessions into Scout

A flat .jsonl log

Every Claude Code session you run leaves an append-only log on disk. scout import claude_code finds those logs, reconstructs the conversation that produced them, and rewrites it in the shape Scout scans and analyzes. This guide walks the whole crossing — what's read, how it maps, and the quiet tricks that make two very different data models line up.

the two sides

Two formats that were never meant to meet.

Claude Code writes a forensic, append-only stream — one JSON object per line, in the order things happened, with no notion of a finished conversation. Scout wants the opposite: a closed, queryable record with messages, a typed event timeline, and roll-up stats. The importer is the bridge between them.

the crossing, step by step

Six stages from log line to Transcript.

The pipeline reads top-to-bottom but isn't a straight pass: it has to rebuild structure the log threw away, then split and merge sessions before any conversion happens.

01
discover

Find the session files

With no path given, the importer scans ~/.claude/projects/ for every project directory, collects the *.jsonl files (skipping the agent-*.jsonl subagent logs), and sorts them newest-first by modification time so --limit can stop early. --from / --to filter by mtime; session_id filters by filename stem.

The directory name is itself encoded data — see the path-decoding trick below.

sources/_claude_code/client.py · discover_session_files
02
parse + consolidate

Read JSONL into typed events

Each line is loaded as a dict, then validated into Pydantic models — UserEvent, AssistantEvent, SystemEvent — by parse_events(). Then consolidate_assistant_events() merges fragments: Claude Code may emit one assistant turn as several lines (a thinking block, then text, then a tool call), all sharing the same message.id. They're folded back into a single assistant event with combined content.

inspect_swe · parse_events · consolidate_assistant_events
03
rebuild structure

Reassemble the conversation tree

The log is flat, but every event carries a parentUuid. build_event_tree() uses those links to reconstruct the branching tree of the session, then flatten_tree_chronological() walks it back into a single ordered sequence — preserving structure while restoring clean chronological order.

sources/_claude_code/transcripts.py
04
split + merge

Decide the transcript boundaries

One file isn't always one transcript, and one transcript isn't always one file. split_on_clear() cuts the event stream at each /clear command — a fresh context deserves its own transcript, so a single file can yield several. In the other direction, plan-mode and execution write separate files that share a slug; those are grouped and later merged into one transcript. (Both are detailed under Tricks.)

transcripts.py · split_on_clear · _merge_transcripts
05
convert

Translate into Scout events

process_parsed_events() is an async generator that turns each parsed event into a Scout Event: an assistant turn becomes a ModelEvent (its text → ContentText, thinking → ContentReasoning, tool_use → tool calls); a tool result becomes a ToolEvent; compaction becomes a CompactionEvent.

Scout's version supplies a file-based agent loader: when it hits a Task tool call, it pulls the subagent's own agent-*.jsonl log and splices that whole sub-conversation in as a nested span — recursively, up to max_depth.

sources/_claude_code/events.py · _load_agent_events
06
assemble

Build the Transcript

From the Scout events, timeline_build() + span_messages(compaction="all") extract the flat message list (subagent messages stay nested, not duplicated). Stable, deterministic IDs are stamped on with stable_message_ids(). Then the roll-ups: model name, summed tokens, and total_time computed as wall-clock minus idle gaps. The finished Transcript is handed to the database.

the parts that bite

Surprises & tricks.

The straight-line mapping above hides the real work. These are the places where the two formats genuinely disagree, and what the importer does about it.

T1

The lossy directory name

Project paths are encoded by replacing every / with - — which is ambiguous the moment a real directory contains a hyphen. The decoder does a filesystem-validated DFS, trying each hyphen as a separator or a literal character, and returns the first interpretation that actually exists on disk.

-Users-my-project-foo →
/Users/my-project/foo  (if it exists)
/Users/my/project-foo  (otherwise)
T2

One turn, many lines

A single assistant turn can be written as several JSONL lines that share a message.id. Left alone they'd become several events. consolidate_assistant_events() merges them by id into one turn; lines without an id pass through untouched.

thinking line + text line + tool_use line
→ one AssistantEvent
T3

Plan and execute, rejoined

Plan mode and the execution that follows are written as two files with different sessionIds but a shared slug. The importer groups by slug and merges them into one transcript — even backfilling a partner file that a time filter would otherwise have hidden — inserting an InfoEvent boundary between phases and recording both UUIDs in metadata.session_ids.

slug "x" · session-aaa  ⟶
slug "x" · session-bbb  ⟶ one Transcript
T4

/clear cuts the tape

A /clear resets the context mid-file. The importer treats each side as its own transcript, splitting the stream on those boundaries and dropping the /clear marker itself. Multi-segment files get suffixed ids like <session>-0, <session>-1.

[ msgs ] /clear [ msgs ]
→ transcript-0  +  transcript-1
T5

Subagents arrive out of order

A Task spawns a subagent whose events live in a separate sidechain log and may stream in before the parent tool call is seen. They're buffered by sessionId and matched FIFO to Task calls; the matching agentId is dug out of the tool result by JSON-parse first, then regex fallback. Team agents — human-readable ids over hashed filenames — are matched by comparing the spawn prompt text against each candidate log.

tool_result → {"agentId":"…"}
load agent-*.jsonl · nest as span
T6

Compaction is kept, not erased

When Claude Code compacts history, that system event becomes a CompactionEvent carrying the trigger (auto/manual) and pre/post token counts. The conversation is still stitched back together — span_messages(compaction="all") reads across the boundary — but the boundary stays in the event timeline for analysis.

system · compact_boundary
→ CompactionEvent{ preTokens, postTokens }
