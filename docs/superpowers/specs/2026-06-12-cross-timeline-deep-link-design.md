# Cross-Timeline Deep-Link Auto-Switch

**Date:** 2026-06-12
**Status:** Approved
**Area:** `src/inspect_scout/_view/ts-mono/packages/inspect-components/src/transcript/`

## Problem

A transcript can have multiple root-level timelines (e.g. `auditor` and
`target`). The visible timeline is selected via the `timeline_view` URL search
param, resolved by `useActiveTimeline`.

Deep links to events and messages (`?event=<uuid>`, `?message=<id>`) are
resolved in `TranscriptLayout` against **only the active timeline's tree**.
Cite links generated by the search panel copy the current URL's params, so
they inherit whatever timeline is currently visible. When a cite (or any
deep link) targets an event that lives in a different root timeline, the
lookup finds nothing and the click silently does nothing. Manually switching
to the containing timeline first makes the same link work.

## Design

When a deep link's target cannot be found in the active timeline, search the
other root timelines; if the target is found in another timeline,
programmatically switch the active timeline to it (equivalent to the user
selecting it in the timeline selector). The existing resolution and
scroll-to-event machinery then runs against the new root and completes the
navigation.

This lives at the resolution layer in `TranscriptLayout` so it covers every
deep-link source: search-panel cites, pasted/shared URLs, and links from
other views. No changes to cite URL generation.

### New helpers

Location: `packages/inspect-components/src/transcript/` (alongside
`resolveMessageToEvent.ts`).

- `findTimelineIndexForEvent(eventId: string, timelines: Timeline[]): number`
  — walks each timeline's root span tree (both `content` and `branches`,
  mirroring the traversal in `collectReferencedIds`) matching `event.uuid`
  against `eventId`. Returns the index of the first containing timeline, or
  `-1`. (`?event=` params are event uuids — see `treeify.ts`, which keys
  event nodes by `event.uuid`.)
- `findTimelineIndexForMessage(messageId: string, timelines: Timeline[]): number`
  — same shape, but per timeline reuses `resolveMessageToEvent` and
  `resolveMessageInBranches` to test for containment.

### TranscriptLayout integration

`TranscriptLayout` already holds `timelines`, `activeTimelineIndex`, and
`setActiveTimeline` (used by the timeline selector in the swimlane header).

Add a memo + effect after the existing message-resolution block
(`resolvedRoot` / its side-effect at ~line 627–664):

- Memo: compute the target timeline index only when a deep link is present
  and unresolved in the active timeline:
  - `initialEventId` set and not found in the active timeline →
    `findTimelineIndexForEvent(initialEventId, timelines)`
  - else `initialMessageId` set and `resolvedLocal`/`resolvedRoot` both
    undefined → `findTimelineIndexForMessage(initialMessageId, timelines)`
  - otherwise no target (`-1`).
- Effect: when the computed index is `>= 0` and differs from
  `activeTimelineIndex`, call `setActiveTimeline(index)`. Guard with the
  same prev-ref pattern as the existing message side-effect so it fires once
  per deep-link change, not on unrelated re-renders (e.g. user manually
  switching timelines away while the stale `?event=` param is still in the
  URL must not snap them back).

After the switch, the URL's `event`/`message` params are untouched;
`timelineData.root` recomputes against the new active timeline, and the
existing `resolvedRoot` / `effectiveInitialEventId` / imperative-scroll flow
completes navigation.

### Edge cases

- Target in the active timeline → no switch; behavior unchanged.
- Target in no timeline → no-op (today's behavior).
- Target in multiple timelines → active timeline wins (no switch); otherwise
  first containing timeline by index.
- `setActiveTimeline` (via `useActiveTimelineSearchParams.onActiveChange`)
  clears the `selected` param — acceptable: a switch only happens when the
  target isn't in the current timeline, so the prior selection is not where
  the user is headed.
- Single-timeline transcripts: helpers never find a different index; no
  behavior change.

## Testing

- Unit tests for `findTimelineIndexForEvent` /
  `findTimelineIndexForMessage`: multi-timeline containment, event inside
  `branches`, not-found, first-match ordering.
- Auto-switch behavior test at the component/hook level following existing
  front-end testing patterns (`apps/scout/design/front-end-testing.md`):
  deep link to an event in a non-active timeline switches `timeline_view`
  and resolves; deep link in the active timeline does not switch; user
  re-switching away with a stale param is not overridden.
- Manual verification against the repro: open a multi-timeline transcript,
  run a grep search producing cites into the non-visible timeline, click a
  cite, confirm the view switches and scrolls to the event.
